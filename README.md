# Event Analytics Pipeline

## Overview

This project implements a **distributed event ingestion and analytics pipeline** inspired by modern telemetry and behavioral analytics systems.

Applications generate large numbers of user interaction events (clicks, page views, logins, etc.). These events must be ingested reliably, buffered under load, processed asynchronously, stored efficiently, and analyzed for insights.

This project simulates that architecture using a scalable backend pipeline built with:

* FastAPI
* Redis
* PostgreSQL
* Docker
* Python workers

The system demonstrates **real-world backend patterns** including event queues, asynchronous processing, caching, and API observability.

---

# Architecture

The pipeline follows a typical **event-driven system design**.

```id="d9u6xv"
Event Generator
      │
      ▼
Ingestion API (FastAPI)
      │
      ▼
Redis Queue
      │
      ▼
Worker Consumer
      │
      ▼
PostgreSQL (JSONB Event Store)
      │
      ▼
Analytics API
      │
      ▼
Redis Cache
      │
      ▼
Client Metrics Queries
```

### Data Flow

1. Events are generated by the **event generator**.
2. Events are sent to the **ingestion API**.
3. The API pushes events into the **Redis queue**.
4. A **worker service** consumes queued events.
5. Events are stored in **PostgreSQL JSONB tables**.
6. The **analytics API** computes metrics from stored events.
7. Frequently requested metrics are cached in **Redis**.
8. Clients request analytics metrics from the API.

# System Components

## Event Generator

Simulates application traffic by generating synthetic user events.

Example events:

* page views
* clicks
* logins
* purchases

The generator continuously sends events to the ingestion API to simulate production traffic.

---

## Ingestion API

The ingestion service is built using **FastAPI**.

Responsibilities:

* Accept event payloads
* Validate request structure
* Push events to Redis queue

The API is designed to remain fast and responsive because it **does not directly interact with the database**.

---

## Redis Queue

Redis acts as the **message queue** between ingestion and processing.

Advantages:

* Handles bursts of traffic
* Decouples ingestion from database writes
* Improves system reliability

Redis databases used:

```
Redis DB 0 → Event Queue
Redis DB 1 → Metrics Cache
```

---

## Worker Consumer

The worker continuously pulls events from Redis.

Workflow:

1. Fetch event from queue
2. Transform payload if needed
3. Store event in PostgreSQL

This asynchronous architecture allows ingestion to scale independently from storage.

---

## PostgreSQL Event Store

Events are stored using **JSONB** to allow flexible event schemas.

Example event structure:

```id="9p0n2n"
{
  "event_id": "uuid",
  "user_id": "user_123",
  "event_type": "page_view",
  "timestamp": "2026-01-01T10:30:00Z",
  "metadata": {
    "page": "/home",
    "device": "mobile"
  }
}
```

Benefits:

* Schema flexibility
* Easy analytics queries
* JSON field indexing

---

## Analytics API

Provides endpoints to compute system metrics.

Endpoints include:

```
GET /metrics/events
GET /metrics/active-users
GET /metrics/events-by-type
```

These queries aggregate event data stored in PostgreSQL.

---

## Redis Metrics Cache

The analytics API uses Redis as a **cache layer**.

```
Client → Analytics API → Redis Cache → PostgreSQL
```

Workflow:

1. Check if metric exists in cache
2. If cached → return instantly
3. If not → query PostgreSQL
4. Store result in Redis
5. Return result

This dramatically reduces database load for repeated queries.

# Metrics Logging

Both APIs include **metrics middleware** to log request information.

Metrics captured:

* endpoint path
* HTTP method
* response status
* request latency

Example log:

```
INFO metrics endpoint=/events method=POST status=200 latency_ms=4.12
INFO metrics endpoint=/metrics/events method=GET status=200 latency_ms=1.43
```

These logs provide visibility into system performance.

# Setup

Clone the repository:

```
git clone <repository-url>
cd event-analytics-pipeline
```

Create a Python virtual environment:

```
python3 -m venv venv
source venv/bin/activate
```

Install dependencies:

```
pip install -r requirements.txt
```

---

# Running the Pipeline

### Start infrastructure

```
docker compose up -d
```

Verify containers:

```
docker ps
```

Expected services:

```
events_postgres
events_queue
```

---

### Start ingestion API

```
uvicorn ingestion_service.main:app --port 8000
```

Interactive documentation:

```
http://localhost:8000/docs
```

---

### Start worker

```
python worker_service/consumer.py
```

---

### Start analytics API

```
uvicorn analytics_api.main:app --port 9000
```

---

### Start event generator

```
python event_generator/generator.py
```

The generator begins sending continuous event traffic.
